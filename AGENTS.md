# quant_v2 Codex 长任务工作流

> 版本: 1.0
> 最后更新: 2026-03-19
> 说明: 量化系统专用 Codex CLI 全局指令，基于 Codex 长任务执行工作流研究

---

## 🎯 设计目标

为量化系统提供简洁、可执行的 Codex 指令。聚焦"做什么"而非"怎么排版"。遵循官方最佳实践：简单、直接、可维护。

**量化系统专项约束**：
- 数据准确性：因子计算 NaN 传播控制、无未来数据泄露（look-ahead forbidden）
- 回测可信度：收益率计算可复现、佣金/滑点正确扣除、随机种子固定
- API 安全：不泄露 JQData 凭证、回测结果注明天赋偏差风险
- 量化术语：代码标识符（变量/函数/类）用英文，注释用中文

---

## 📊 优先级栈

当规则冲突时，按以下优先级执行（从高到低）：

1. **安全与合规**：不泄露 API 凭证、不引入未来数据、回测注明天赋偏差、破坏性操作先说明
2. **CSV 唯一状态源**：只读写用户传入的那一份 CSV，禁止新建汇总 CSV
3. **pytest 闭环**：能跑测试就跑，受限验收必须落盘 `notes`
4. **KISS/YAGNI**：不做无关重构，优先修根因，保持向后兼容性
5. **报告规范**：关键文件引用 `path:line`，显式列出风险和后续步骤

## 🔧 Plan 模式

🎯 **使用场景**：
- 适用：多步骤、跨模块、涉及架构/测试设计的任务
- 不适用：单文件小改动、一次性问答

🔧 **入口**：`/prompts:plan <任务描述>`

🧠 **复杂度分级（量化专用）**：
- `single-factor`：单因子调参/IC验证（3-5步）
- `multi-factor`：多因子组合/权重优化（5-8步）
- `system`：全链路重构/新模块（8-12步）

📁 **Plan 文件规范**（`plan/*.md`）：
- 目录：`plan/`
- 命名：`YYYY-MM-DD_HH-mm-ss-<slug>.md`
- 头部元数据（YAML frontmatter，必须）：

```markdown
---
mode: plan
cwd: F:\github\quant_v2
task: <任务标题>
complexity: <single-factor|multi-factor|system>
planning_method: builtin
created_at: <ISO8601>
---
```

## 💻 代码规则（量化专项）

### 通用原则

- **KISS / YAGNI**：简单直接，不为假设需求过度设计
- **单一职责**：函数做一件事，嵌套控制在 3 层内
- **向后兼容**：不经批准不破坏现有 API/CLI 行为/数据格式
- **量化专项**：

| 规则 | 说明 |
|------|------|
| NaN 传播控制 | 因子计算函数必须在返回前 dropna 或明确说明 NaN 位置 |
| look-ahead 禁止 | 禁止使用未来数据（`.shift(-n)` 用于计算 forward returns 时仅限验证层） |
| 随机种子固定 | 涉及随机数的测试/优化必须设置 seed |
| 收益率计算 | 必须用 `pct_change()` 或对数收益率，禁止手工复利计算 |
| 缓存安全 | JQData 缓存 TTL 默认 7 天，超出后必须重新验证 |

## 🛠️ 工具约定

### Shell 与文件系统

- 默认通过 Codex CLI 执行命令
- **读多写少**：优先只读命令
- 避免破坏性命令（`rm -rf`、强制覆盖），除非明确授权
- 大范围操作先小范围试验

### 测试工具

| 工具 | 用途 | 命令 |
|------|------|------|
| `pytest` | 单元测试 | `python -m pytest tests/ -v --tb=short` |
| `pytest` | 模块测试 | `python -m pytest tests/test_<module>.py -v` |
| `mypy` | 类型检查 | `python -m mypy src/` |
| `ruff` | 代码检查 | `python -m ruff check src/` |

---

## 🔒 安全与合规

- 不访问或泄露敏感信息（API 密钥、凭证、个人数据）
- 破坏性操作前说明影响范围，获得确认再执行
- 回测结果必须标注"历史业绩不代表未来收益"
- JQData 凭证不得写入代码，使用环境变量或配置文件分离

## ✅ 实施检查清单（每条 issue 闭环前必须自检）

- [ ] 接触工具前已记录"接收与现实检查"
- [ ] pytest 测试通过（无 failed）
- [ ] 因子计算 NaN 处理正确（warmup 期间合理）
- [ ] 回测引擎关键路径有测试覆盖
- [ ] CSV 状态字段已全部更新为枚举值
- [ ] git commit 包含代码 + CSV（同一 commit）
- [ ] `plan/*.md` 元数据完整（frontmatter）

## 🔁 循环协议

每轮小循环的完整链路：

```
1. 用户提出需求 → /prompts:plan
2. 人工走读 Plan → 补充测试/验收口径
3. /prompts:plan_to_issues_csv → issues/YYYY-MM-DD_*.csv
4. 新会话 /prompts:issues_csv_execute <csv>
   └→ 逐条闭环（实现→pytest→integration→Review→commit）
5. /prompts:regression
   └→ 全量回归 → 0 failed
6. 回到步骤1（下一轮新需求）
```

**元数据维护**：`issues/.meta.csv` 记录每轮轮次/日期/commit hash/完成数
